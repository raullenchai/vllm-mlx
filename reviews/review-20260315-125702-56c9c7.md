## High

### 1) `vllm_mlx/mllm_scheduler.py:744`
- Severity: high
- Category: Diff
- Description: `_process_loop()` now runs `_step_no_queue()` on a worker thread whenever `self.waiting` is non-empty, but request admission still mutates `self.requests` and `self.waiting` directly on the event-loop thread in `add_request()` (`vllm_mlx/mllm_scheduler.py:320`). That leaves the same `dict`/`deque` state being mutated from two threads without any lock or single-thread handoff, so concurrent request arrivals/aborts can race with scheduling and produce dropped work, stale bookkeeping, or intermittent queueing bugs.
- Suggested fix: Keep all scheduler-state mutation on one thread and offload only the blocking vision/model work, or add an explicit synchronization boundary around `waiting`, `running`, `requests`, and the UID maps. Add an async regression test that submits/aborts while a prefill step is executing.

## Medium

### 2) `vllm_mlx/models/mllm.py:1614`
- Severity: medium
- Category: Diff
- Description: The recent MLLM cache fixes land in hot-path logic, but there are no new regression tests covering them. In particular, this stream-cache path now depends on `chunk.prompt_tokens` for multimodal correctness and zero-token handling, yet no adjacent test exercises an empty stream, missing `prompt_tokens`, or streamed multimodal cache reuse. The same gap applies to the nearby scheduler/batch-generator fixes in this review window.
- Suggested fix: Add focused tests for streamed multimodal cache storage/reuse, zero-token streams, deferred-abort processing during executor work, and prompt-cache extraction before batch filtering.

### 3) `vllm_mlx/scheduler.py:44`
- Severity: medium
- Category: Diff
- Description: The cache-corruption matcher is narrower than before, but the raw substring `"KVCache"` is still broad enough to match non-corruption exceptions that merely mention the cache type. Because `_is_cache_corruption_error()` does a simple `pattern in str(error)` check, false positives still trigger expensive cache clears and request rescheduling for unrelated failures.
- Suggested fix: Replace class-name substrings with more specific signatures or exception-type checks, and add a regression test proving that benign `KVCache`-mentioning errors do not enter cache-recovery.

### 4) `vllm_mlx/mllm_scheduler.py:739`
- Severity: medium
- Category: Diff
- Description: `_process_loop()` creates a `ThreadPoolExecutor`, but `stop()` only cancels the task and closes the batch generator; it never shuts the executor down. Repeated start/stop cycles will leak a worker thread per scheduler instance and can make tests or long-lived processes harder to cleanly tear down.
- Suggested fix: Store the executor on `self` and call `shutdown(wait=False, cancel_futures=True)` from `stop()` or a `finally` block inside `_process_loop()`.

### 5) `.`
- Severity: medium
- Category: Holistic
- Description: The repository has no checked-in `AGENTS.md` or `CLAUDE.md` at the root or in major subtrees. For a codebase explicitly being reviewed and edited with agents, that means there is no authoritative guidance on module ownership, safe commands, validation expectations, naming conventions, or high-risk areas.
- Suggested fix: Add a root `AGENTS.md` plus matching `CLAUDE.md`, then add scoped agent docs for major areas such as `vllm_mlx/`, `tests/`, `docs/`, and `scripts/` that cover boundaries, common pitfalls, and the minimum validation matrix.

### 6) `vllm_mlx/server.py:1`
- Severity: medium
- Category: Holistic
- Description: Core runtime logic is concentrated in a few very large modules. `vllm_mlx/server.py` is over 3200 lines, `vllm_mlx/scheduler.py` is over 2500 lines, and `vllm_mlx/models/mllm.py` is nearly 1900 lines. That makes it hard for a new developer or agent to quickly find the correct layer to edit, and it increases the chance that changes mix transport, orchestration, caching, and model-specific logic.
- Suggested fix: Split these modules by responsibility into smaller routers/services/helpers, with clearer boundaries between API transport, scheduler orchestration, cache management, and model adapters.

## Low

### 7) `vllm_mlx/cli.py:145`
- Severity: low
- Category: Holistic
- Description: Environment/config handling is still spread across CLI parsing, direct `os.environ` mutation, startup hooks, and prose documentation. For example, the MCP config path is set in `vllm_mlx/cli.py:145` and later read in `vllm_mlx/server.py:333`, while the documented environment surface lives separately in `docs/reference/configuration.md:116`. That split makes startup behavior less discoverable for contributors and agents, and there is no checked-in `.env.example` to anchor the supported variables.
- Suggested fix: Introduce a single typed settings layer that resolves CLI flags and environment variables in one place, validate it at startup, and add a minimal `.env.example` or equivalent machine-readable config sample.

### 8) `vllm_mlx/`
- Severity: low
- Category: Holistic
- Description: The project has solid logging and a useful `/v1/status` endpoint, but there is little structured observability around the highest-risk runtime paths such as scheduler contention, cache recovery, abort handling, and multimodal prefill latency. That makes production debugging and performance regression analysis harder than it needs to be.
- Suggested fix: Add lightweight counters/timers around scheduler steps, cache hits/recoveries, abort paths, and prefill/generation latency, then expose them through a metrics endpoint or structured logs.
