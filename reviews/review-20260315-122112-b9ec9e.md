## High

### 1) `vllm_mlx/mllm_scheduler.py:685`
- Severity: high
- Category: Diff
- Description: `_process_loop()` now offloads the entire `self.step()` call to `run_in_executor()` whenever there are waiting requests. `step()` mutates shared scheduler state (`waiting`, `running`, `requests`, UID maps) and also writes to `asyncio.Queue` instances, but those structures are not thread-safe across the event-loop thread and a worker thread. Concurrent `add_request_async()`, abort/cleanup paths, or streaming consumers can race with the worker thread and cause stale bookkeeping, lost outputs, or intermittent hangs.
- Suggested fix: Keep scheduler bookkeeping and queue delivery on the event-loop thread, and offload only the blocking vision/prefill work. If background execution is unavoidable, add explicit synchronization around shared state and route queue writes through `loop.call_soon_threadsafe(...)`.

### 2) `vllm_mlx/models/mllm.py:1616`
- Severity: high
- Category: Diff
- Description: The new streaming cache-store path derives `prompt_tokens_count` from `self.processor.tokenizer.encode(formatted_prompt)`. For multimodal prompts, the real KV-cache length includes image/video-expanded prompt tokens, so plain text tokenization can undercount the cached prefix. The subsequent `:prompt_tokens_count` slice can therefore store a truncated KV cache and poison later streamed cache hits for image/video requests.
- Suggested fix: Derive the prompt length from the actual multimodal generation metadata (`result.prompt_tokens`, chunk metadata, or the prompt-cache offset before generation) instead of plain text tokenization. Add a regression test that reuses a streamed multimodal cache entry.

### 3) `vllm_mlx/mllm_scheduler.py:489`
- Severity: high
- Category: Diff
- Description: Stop-sequence trimming currently clears `output.new_text` for the whole final chunk as soon as the stop string appears anywhere in `decoded_so_far`. If the stop marker begins in the middle of a decoded token fragment, clients that assemble streamed output from `new_text` lose the valid prefix that preceded the stop marker and receive truncated output.
- Suggested fix: Compute the chunk-local prefix before the stop marker and emit that prefix in `output.new_text`, while suppressing only the suffix at or after the matched stop string. Add a regression test where the stop sequence begins mid-token.

### 4) `README.md:1`
- Severity: high
- Category: Holistic
- Description: The repository is in a partial rename state. Public docs market the project as `Rapid-MLX` and the `rapid-mlx` CLI, but packaging metadata, console entry points, installer URLs, and much of `docs/` still identify the project as `vllm-mlx`. That creates two competing canonical identities for the same codebase, which makes onboarding, automation, packaging, and agent-driven edits error-prone.
- Suggested fix: Pick one canonical product/package/CLI identity and update `README.md`, `pyproject.toml`, `install.sh`, docs, URLs, and examples together. If compatibility aliases must remain, document them explicitly and test both names.

### 5) `.:1`

### 6) `vllm_mlx/server.py:1`
- Severity: high
- Category: Holistic
- Description: Core runtime responsibilities are concentrated in a few oversized modules instead of clear service boundaries. `vllm_mlx/server.py` is over 3200 lines, and `vllm_mlx/scheduler.py`, `vllm_mlx/benchmark.py`, and `vllm_mlx/cli.py` are also very large. API routing, auth/config handling, model lifecycle, MCP wiring, embeddings, tool parsing, routing, and temp-file handling are spread across these top-level modules, which makes changes harder to reason about and increases regression risk.
- Suggested fix: Split runtime code by responsibility into narrower modules: API routers, request/response adapters, model lifecycle services, feature services, and typed configuration/loading layers. Reduce module-global state and move feature-specific logic into subpackages with smaller interfaces.

## Medium

### 7) `vllm_mlx/mllm_scheduler.py:504`
- Severity: medium
- Category: Diff
- Description: Finalization uses `if not request.output_text` to decide whether stop-string trimming happened. When a stop string matches at position `0`, the correctly trimmed result is the empty string, so this branch re-decodes the full token sequence and leaks the stop text back into `output.output_text`.
- Suggested fix: Track stop-string trimming with an explicit boolean or sentinel instead of string truthiness, or keep `output_text` nullable until finalization. Add a regression test for a stop sequence that matches at the start of generation.

### 8) `vllm_mlx/mllm_batch_generator.py:27`
- Severity: medium
- Category: Diff
- Description: `make_sampler` is now imported at module import time. That bypasses the package's lazy-import strategy described in `vllm_mlx/__init__.py:17` and can reintroduce import-time failures in environments where `mlx` is available but `mlx_lm` is not, even before multimodal batching is exercised.
- Suggested fix: Move the `make_sampler` import back into the execution paths that require it, or guard it with a fallback and a clearer error path. Add an import-only regression test for `vllm_mlx.mllm_batch_generator` when `mlx_lm` is unavailable.

### 9) `docs/development/architecture.md:111`
- Severity: medium
- Category: Holistic
- Description: The architecture documentation is stale enough to mislead maintainers and agents. It references modules that do not exist (`reasoning_parsers/`, `vllm_mlx.hardware`), omits current structure such as `vllm_mlx/engine/hybrid.py`, and understates parser coverage relative to the current README. That weakens the repo's navigability for both humans and automated contributors.
- Suggested fix: Update the architecture docs to mirror the current package tree and command surface, and add a lightweight CI check or generated inventory so module names and feature counts stay aligned with the codebase.

### 10) `docs/development/contributing.md:35`
- Severity: medium
- Category: Holistic
- Description: Contributor workflow documentation is not single-sourced. The docs still recommend `black` and `isort`, while CI is configured around `ruff`, and version metadata is inconsistent between `pyproject.toml` (`0.2.6`) and `vllm_mlx/__init__.py` (`0.2.5`). This kind of tooling drift makes the expected development workflow ambiguous.
