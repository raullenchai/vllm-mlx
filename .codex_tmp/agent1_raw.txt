- File: `vllm_mlx/mllm_scheduler.py:685`
- Severity: high
- Category: Diff
- Description: `_process_loop()` now sends the entire `self.step()` through `run_in_executor()`. `step()` mutates shared scheduler state (`waiting`, `running`, `requests`, UID maps) and also writes into `asyncio.Queue` objects, but neither that state nor `asyncio.Queue` is thread-safe across the event-loop thread and a worker thread. Concurrent `add_request_async()`, `stream_outputs()` cleanup, or aborts can race and cause lost outputs, stale bookkeeping, or intermittent hangs.
- Suggested fix: Keep scheduler bookkeeping and queue delivery on the event-loop thread and offload only the blocking model/prefill work to a worker. If you keep the thread hop, add explicit synchronization and use `loop.call_soon_threadsafe(...)` for queue operations.

- File: `vllm_mlx/models/mllm.py:1616`
- Severity: high
- Category: Diff
- Description: The new streaming cache-store path derives `prompt_tokens_count` from `self.processor.tokenizer.encode(formatted_prompt)`. For multimodal prompts, the actual KV length includes image/video-expanded prompt tokens, so this count can be smaller than the real prompt cache length. The subsequent slice at `:prompt_tokens_count` can store a truncated/corrupt KV cache, breaking later cache hits for streamed image/video requests.
- Suggested fix: Derive prompt length from actual multimodal processing metadata (`processed.num_tokens`, `chunk.prompt_tokens`, or the cache offset before generation), not plain text tokenization. Add a regression test that reuses a streamed multimodal cache entry.

- File: `vllm_mlx/mllm_scheduler.py:489`
- Severity: high
- Category: Diff
- Description: When a stop string is found, the code clears `output.new_text` for the whole chunk. If the stop marker starts in the middle of a token decode (for example a token decodes to `ld###`), streaming clients that assemble output from `new_text` lose the valid prefix before the stop marker and end up with truncated text.
- Suggested fix: Emit the non-stop prefix from the final chunk in `new_text` and only suppress the suffix at/after the stop marker. Add a regression test where the stop string begins mid-token.

- File: `vllm_mlx/mllm_scheduler.py:504`
- Severity: medium
- Category: Diff
- Description: The finalization path uses `if not request.output_text` to decide whether the output was trimmed by stop-string handling. If the stop string occurs at position 0, the trimmed result is the empty string, so this branch re-decodes the full token sequence and leaks the stop text back in `output.output_text`.
- Suggested fix: Track stop-string trimming with an explicit boolean/sentinel instead of relying on string truthiness, or make `output_text` nullable until finalization. Add a regression test for a stop sequence at the very start of generation.
